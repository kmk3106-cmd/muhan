# 무한매수법 - T값 산출 로직 및 이슈 분석

## 1. 3월 9일 미거래일 인식 이슈

### 1.1 차이점: 무한매수법 vs 떨사오팔

| 항목 | 무한매수법 | 떨사오팔 |
|------|-----------|----------|
| **거래일 체크** | `is_us_trading_day()` 사용 | **체크 없음** |
| **스케줄 시** | 거래일이 아니면 **전체 스킵** (동기화 포함) | 항상 실행 |
| **동작** | 토/일(ET) → 워커 미실행 | 토/일이어도 워커 실행 |

### 1.2 `is_us_trading_day()` 로직

```python
# worker.py
def is_us_trading_day() -> bool:
    """미국 거래일 여부 - 미국 동부(ET) 시간 기준 월~금"""
    et = datetime.now(ZoneInfo("America/New_York"))
    return et.weekday() < 5  # 0=월 ~ 4=금
```

- **미국 동부(ET)** 기준으로 `weekday()`만 확인
- NYSE 휴장일(예: 신년, MLK Day 등)은 **반영 안 함**
- **서머타임(DST)** 은 `ZoneInfo("America/New_York")`으로 자동 반영됨

### 1.3 3월 9일과 DST

- **2025년 3월 9일**: 미국에서 DST 시작일(일요일 새벽 2시)
- **2025년 3월 9일(ET)** = 일요일 → `weekday() = 6` → **미거래일**
- 따라서 3월 9일(ET)에는 워커가 스킵되는 것이 **정상 동작**임

### 1.4 의심 가능 시나리오

1. **서버/클라이언트 시간대 혼선**
   - 서버가 UTC 등 다른 타임존이면, KST 3월 10일 00:00 실행 시 ET는 3월 9일 11:00 (일요일)
   - 이 시점에 실행되면 `is_us_trading_day()` = False → 스킵

2. **날짜 해석 차이**
   - 사용자는 “3월 9일 거래”를 기대했을 수 있으나, ET 기준 3월 9일은 일요일로 실제 거래일 아님

3. **떨사오팔이 3월 9일 거래했다는 의미**
   - 떨사오팔은 거래일 체크가 없어 **스케줄 시점에는 항상 실행**
   - 3월 9일(ET) 일요일에는 미국 장이 닫혀 있어, 실제 체결은 나지 않고 주문만 제출·대기 상태였을 가능성 있음

### 1.5 개선 제안

- NYSE 공식 휴장일 반영: `_us_market_holidays()` (ddsop에 있는 로직)와 동일하게 휴장일 체크 추가
- 거래일 기준 명확화: “오늘 실행해서 오늘 장에서 거래”가 아니라, “미국 장이 열리는 날에만 실행”이 목적이면 `is_us_trading_day()` 유지하고, 로그/대시보드에 “미거래일 스킵” 사유를 표시

## 2. 거래내역(체결) 미표시 이슈

### 2.1 체결 기록 경로

체결은 **워커 실행 시** `_record_executions()` 에서 KIS 체결 API(`inquire_ccnl`) 결과를 읽어 `trades` 테이블에 기록합니다.

```text
_process_portfolio()
  → inquire_ccnl(pdno=portfolio.ticker, ord_strt_dt=90일전, ord_end_dt=today)
  → sync_state_from_api(..., ccnl_df)
  → _record_executions(session, portfolio, today, ccnl_df)
```

### 2.2 거래내역에 안 뜨는 경우

| 원인 | 설명 |
|------|------|
| **워커 미실행** | `is_us_trading_day()` = False 이면 전체 스킵 → `_record_executions()` 호출 안 됨 |
| **ccnl 조회 구간** | `ord_strt_dt` = 90일 전, `ord_end_dt` = today(KST) |
| **포트폴리오 등록일** | `portfolio.created_at` 이전 체결은 무시 |
| **odno 비어 있음** | KIS 응답에서 odno가 없으면 해당 건 스킵 |
| **컬럼명 불일치** | KIS 응답 필드명(sll_buy_dvsn_cd, ord_dt 등)이 예상과 다르면 파싱 실패 가능 |

### 2.3 3월 10·11일 체결이 안 보이는 경우

1. **3월 9일(ET) = 일요일**이라 워커가 스킵됨  
   → 3월 10일(월) 워커가 실행되면 그때 ccnl을 조회해 3월 10·11일 체결이 기록됨.

2. **워커가 연속으로 스킵된 경우**
   - 서버 점검, Kill Switch, 인증 실패 등으로 며칠 동안 워커가 안 돌면, 체결 동기화가 뒤로 밀림  
   - 한 번이라도 정상 실행되면, 90일 구간 내 체결은 그때 기록됨.

3. **수동 동기화**
   - 대시보드 **“동기화만”** 버튼은 `force=True` 로 워커를 돌려서, 거래일 체크를 건너뜀  
   - 이때도 `_record_executions()` 가 호출되므로, 누락된 체결을 나중에라도 불러올 수 있음.

## 3. T값 산출 로직 (상세)

### 3.1 기본 공식

```
T = (매수누적액 - 매도누적액) / 1회매수액(B)
  = (cum_buy_amount - cum_sell_amount) / B
```

- **cum_buy_amount**: 현재 싸이클 내 누적 매수금액
- **cum_sell_amount**: 현재 싸이클 내 누적 매도금액
- **B**: 1회 매수금액 (포트폴리오 설정)

### 3.2 반올림 규칙

- 소수 둘째자리에서 **올림** 후 소수 첫째자리까지 표기  
  - 예: 22.81 → 22.9, 22.01 → 22.1  
- 구현: `math.ceil(raw * 10) / 10`

### 3.3 매수 시 T 변화

| 상황 | cum_buy | cum_sell | T 변화 |
|------|---------|----------|--------|
| 매수 $100 체결 | +$100 | 0 | T 증가 (예: 22.0 → 23.0) |
| 매수 $200 체결 | +$200 | 0 | T 2 단위 증가 |

예: B=$100, 기존 T=22.0 (cum_buy=$2200, cum_sell=$0)  
→ $100 추가 매수 시 net = $2300 → T = 23.0

### 3.4 매도 시 T 변화

| 상황 | cum_buy | cum_sell | T 변화 |
|------|---------|----------|--------|
| 매도 $250 체결 | 유지 | +$250 | T 감소 |
| 1/4 LOC 매도 | 유지 | 증가 | T 감소 |
| 3/4 LIMIT 매도 | 유지 | 증가 | T 감소 |
| MOC 매도 | 유지 | 증가 | T 감소 |

예: B=$100, 기존 T=25.0 (cum_buy=$2500, cum_sell=$0)  
→ $200 매도 시 net = $2300 → T = 23.0

### 3.5 T값이 사용되는 곳

| 용도 | 조건 | 설명 |
|------|------|------|
| **전반전/후반전** | T < 20 / T ≥ 20 | NORMAL 모드 내 구간 |
| **QUARTER 진입** | 39.1 ≤ T ≤ 40 | 원금 소진 구간 → QUARTER 모드 |
| **☆% 계산** | `☆% = R - (T/2) × (40/A)` | LOC 매수/매도 목표가 결정 |

### 3.6 데이터 출처 (sync_state_from_api)

1. **우선: 잔고 API `frcr_pchs_amt1`**
   - `balance_buy_amt > 0` 이고 `qty > 0` 이면  
     `cum_buy_amount = balance_buy_amt`, `cum_sell_amount = 0` 으로 설정  
   - 체결 90일 제한과 무관하게 보유 종목 매입금액 기준 T 계산

2. **차선: 체결(ccnl) 합산**
   - ccnl에서 매수/매도 체결금액을 합산해 cum_buy, cum_sell 계산  
   - `cycle_start_date` 또는 `created_at` 이후 체결만 사용

3. **하한 보정**
   - cum_buy < 평단×수량 이면 cum_buy = 평단×수량  
   - 90일/데이터 누락 대비

4. **상한 보정**
   - cum_buy > 평단×수량×1.5 이고 cum_sell=0 이면 비정상으로 보고  
     cum_buy = 평단×수량, cum_sell = 0 으로 보정
