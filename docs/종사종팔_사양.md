# 종사종팔법 — 구현 사양서 (확정본 · 미구현)

> 상태: **플랜 확정, 코드 미구현.** 사용자 최종 승인(P6) 전까지 어떤 코드도 작성/배포하지 않는다.
> 작성 근거: 사용자 Q&A 확정(2026-05). 종사종팔 = ddsop(떨사오팔) 골격 + **매수부 한 곳**만 교체.

---

## 0. 한 줄 요약

**종사종팔 = "종(가)사(고) 종(가)팔(고)"** — 매 거래일 종가에 무조건 MOC 매수, 각 트렌치는
자기 매수평단 +목표% 도달 시 LOC 익절(목표가 보장), 40거래일 내 미도달 시 MOC 손절.
**떨사오팔과 매도·손절·트렌치·싸이클은 100% 동일하고, 매수 방식만 다르다.**

---

## 1. 확정 사양

| 요소 | 종사종팔 | 떨사오팔(ddsop) 대비 |
|---|---|---|
| 트렌치 | num_tranches 분할, `find_next_buy_tranche` 순서(손절 공석 건너뛰기 포함) | **동일** |
| **매수** | **매 거래일 다음 트렌치 1개 `MOC 무조건` 매수.** 수량 = `floor(트렌치금액 / 전일종가)`. 빈 트렌치 없으면 매수 안 함 | **🔴 유일한 차이** (ddsop는 `LOC at 전일종가×(1−x%)` 조건부) |
| 익절 매도 | BOUGHT & `days_held < 40`: `LOC at 평단×(1+목표%)`. 종가가 목표 이상일 때만 종가 체결 → **목표 미달 매도 없음** | **동일 메커니즘** |
| 손절 매도 | `days_held ≥ 40`: `MOC 시장가 매도` (무조건 종가 청산) | **동일** |
| 목표 수익률 | **종목별 단일 목표%**, 각 트렌치 **자기 매수평단** 기준. 기본값 **3.5%** | ddsop `x_pct` 자리지만 **매도에만** 사용 |
| 손절 기준 | **40거래일** (`loss_cut_days`, 기본 40 · 종목별 설정 가능) | **동일** |
| 매수 빈도 | **매 거래일 1트렌치** (한 번에 여러 트렌치 아님) | **동일** |
| 트렌치 수 | **사용자 지정, 기본 7트렌치** | ddsop 기본 5 |
| 운용 종목 | 사용자 지정, **타 전략과 절대 무중복**(ticker_registry 강제) | 동일 |
| 금액 할당 | 전략별·종목별 **사용자가 직접 지정** | 동일 |
| 손절 후 재매수 | 40일 손절로 비워진 트렌치는 **다음날 다시 MOC 매수로 재진입**(싸이클 계속) | **동일** |
| 싸이클 | T1 매도완료 시 종료 → 재시작. 손절은 싸이클 종료 아님 | **동일** |

### 핵심 정정 메모 (Q&A 중 확인)
- 사용자가 처음 고른 "MOC 매도"는 **40거래일 손절**을 의미했고, **익절 매도는 목표가 보장(LOC)** 가 맞다.
- 즉 "40일 안에는 목표 밑이면 절대 매도하지 않고, 40일 지나면 그때 MOC 손절" → ddsop 매도 로직과 동일.

## 2. 파라미터 의미 변화 (ddsop → jongsa)

- ddsop `x_pct`: 매수임계 & 매도목표 **공용** → jongsa에선 **매도 목표% 단독**(매수엔 % 없음). 기본값 3.5%.
- UI 라벨: "변동률 x%" → **"목표 수익률 %"** 로 변경.
- `seed_reflect_enabled`(추가입금 반영), `total_usd`(종목 시드), `num_tranches`, `loss_cut_days` 는 ddsop 그대로 계승.

## 3. 신규 패키지 `strategies/jongsa/` (ddsop 복제)

운영중 ddsop 무변경 원칙 → **신규 패키지로 복제 후 매수부만 수정**.

```
strategies/jongsa/
  config.py        # DATABASE_URL = .../jongsa.db, KILL_SWITCH_FILE = .../jongsa/.kill_switch
  models.py        # ddsop 동일 스키마 (Ticker/Tranche/Trade/TradeOrder/CycleHistory/AppLog/SystemConfig)
  kis_client.py    # ddsop와 바이트동일 (검증된 코어 — 무수정 복제, reverse-diff로 검증)
  trading_logic.py # ★ generate_orders 매수부만 수정 (§3.1)
  worker.py        # ddsop 복제. generate_orders 호출부 동일. prev_close는 '수량 산출용'으로만 사용
  main.py          # 서브앱(라우트 동일). 스케줄 레인만 jongsa 값(§5)
  settings_store.py, simulate.py, dashboard.html  # 복제
```

### 3.1 `trading_logic.generate_orders` 매수부 교체 (유일한 로직 변경)

```python
# (ddsop 원본)
#   buy_price = round(prev_close * (1 - x/100), 2)
#   order_type = "LOC"

# (jongsa) ↓ 무조건 MOC, % 조건 제거
next_t = find_next_buy_tranche(tranches)
if next_t and prev_close > 0:
    buy_qty = max(1, int(amt_per / prev_close))   # 수량은 전일종가로 산출(추가 KIS 호출 0)
    cy = getattr(ticker_obj, "current_cycle", 1) or 1
    orders.append(OrderItem(
        ticker=symbol, tranche_id=next_t.id, tranche_num=next_t.tranche_num,
        cycle_number=cy, side="buy", order_type="MOC",
        price=0.0, qty=buy_qty, desc="종가 MOC 무조건 매수",
    ))
# 매도/손절부는 ddsop 그대로:
#   - BOUGHT & days_held<40 : LOC at 평단×(1+목표%)
#   - BOUGHT & days_held≥40 : MOC 손절
```

- `amt_per` 산출(시드반영 OFF/ON 분기)·`find_next_buy_tranche`·매도부·손절부는 ddsop 코드 그대로.
- jongsa에서 `x_pct` 필드는 **매도 목표%로만** 쓰이고 매수 임계엔 미사용.

### 4.1 프론트엔드 (전용 반영 — "자동 확장"에 의존하지 않음)

ADAPTERS 기반으로 일부(KPI 합산·전략목록·총자산)는 자동 반영되지만, **종사종팔 전용 UI를 명시적으로 구현**한다:

| 화면 | 종사종팔 전용 반영 |
|---|---|
| 대시보드 | 전략 목록/KPI/도넛/막대바에 종사종팔 포함. 막대바 색상 팔레트 3번째 슬롯 배정 |
| 전략 관리 | **종사종팔 종목 등록/수정 폼**: 라벨 "**목표 수익률 %**"(기본 3.5), 트렌치 수, 종목 시드, 손절일(40), 시드반영 토글. ddsop의 "변동률 x%" 문구는 종사종팔에선 "목표 수익률 %"로. 매수 방식 안내문 "매 거래일 종가 MOC 무조건 매수" 표기. 진행 ON/OFF 토글 |
| 포트폴리오 | **트렌치 현황 시각화**(ddsop계열과 동일 — `/jongsa/api/tickers/{id}/tranches`), 트렌치별 상태/평단/목표가/보유일 |
| 주문/체결 | 종사종팔 주문이 **MOC 매수 / LOC 익절 / MOC 손절**로 정확히 표시. 체결내역 검색·정렬(전략 드롭다운에 종사종팔 추가)·싸이클/회차 컬럼 |
| 성과 분석 | 종사종팔 싸이클별 손익·매수/매도 상세, 전략 실현성과 요약 행 추가 |
| 리스크/모니터링 | 종사종팔 상태·오류로그·Kill Switch 표시 |
| 시스템 설정 | Kill Switch 토글에 종사종팔 행 추가. 감시기능 cycle_integrity에 종사종팔 결과 표출 |

- **백엔드(서브앱 API) + 프론트(통합 SPA)** 양쪽 모두 산출물에 포함. 어느 한쪽만 반영하지 않는다.

## 5. 스케줄 레인 (단일계좌 동시호출 방지 — 확정)

| 전략 | 워커 | sync 초 | pre_auth |
|---|---|---|---|
| infinite | run_m :00 | 홀수분 :20 | −5분 |
| ddsop | 개장+30분 | 짝수분 :40 | −3분 |
| **jongsa** | **개장+40분** | **짝수분 :10** | **−4분** |

- 비중복 확인: 짝수분엔 ddsop :40 / jongsa :10(30초 격차), 홀수분엔 infinite :20만. 어느 (분,초)에도 호출 전략 ≤1.
- 워커는 ddsop 개장+30, jongsa 개장+40 → 10분 격차. MOC는 종가 체결이라 개장+40분 제출도 마감 한참 전이라 안전.
- `schedule_map.py`에 `JONGSA_WORKER_MIN_OFFSET`, `JONGSA_SYNC_MINUTE="0-58/2"`, `JONGSA_SYNC_SECOND=10`, `JONGSA_PRE_AUTH_LEAD_MIN=4` 추가.

## 6. 구현 단계 (사용자 승인 후 진행 — 각 단계 검증 게이트)

- **P1** 패키지 복제 + 바이트동일 검증(kis_client/worker reverse-diff로 코어 보존 입증)
- **P2** `generate_orders` 매수부 MOC 교체 + `simulate.py` 단위 시뮬: 매수(매일 1트렌치 MOC)/익절(LOC 목표가 보장)/손절(40일 MOC)/싸이클 정산 확인
- **P3** 부모 백엔드 통합(mount·adapters·registry·schedule·killswitch·deploy·t_audit)
- **P3.5** **프론트엔드 전용 반영**(§4.1): 전략관리 등록폼·트렌치 시각화·주문/체결·성과·시스템설정 종사종팔 대응. 백/프론트 양쪽 완결
- **P4** strategy-logic-validator 게이트: 무조건 MOC 매수 / 목표가보장 LOC 익절 / 40일 MOC 손절 / 단일계좌 티커귀속 / 레인분리 검증
- **P5** 스테이징 드라이런(3레인 동시 트리거, MOC 수량 산출 실측, **프론트 화면 실측**)
- **P6** 운영 반영(**사용자 최종 승인 필수**) — Kill Switch OFF로 신규 티커 등록·시드배분 후 가동

## 7. 불변식·주의

- **티커 무중복**: infinite/ddsop/jongsa 활성 티커는 절대 겹치지 않음(ticker_registry 강제).
- **단일 계좌(69567573) 공유**: 토큰 파일 캐시 공유, 스케줄 레인 분리로 동시호출 회피.
- **검증된 코어 무변경**: kis_client/worker의 KIS 호출 메커니즘은 복제만, 수정은 trading_logic 매수부 한 곳.
- **MOC 매수 수량 오차**: 전일종가로 수량 산출 → 실제 체결은 종가라 금액에 소폭 오차 가능(허용).

## 8. 확정된 사용자 답변 (근거)

1. 매수 빈도: **매 거래일 1트렌치**
2. 손절 후 재매수: **동일**(다음날 MOC 재진입, 싸이클 계속)
3. 목표 수익률 기본값: **3.5%**
4. 표시명: **종사종팔** (폴더 `strategies/jongsa`) — "종자종팔"은 오타로 간주
5. 스케줄 레인: 개장+40분 / 짝수분:10초 / pre_auth −4분 (타 전략과 비중복 확인)
